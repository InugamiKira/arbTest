# LOF基金套利监控看板 (JSL 风格)

## 📋 项目概述

本项目是一个**高仿集思录 (JSL) 风格**的 LOF/ETF 基金套利监控看板系统，用于实时获取 LOF 基金的净值、收盘价、申赎状态、手续费等数据，并生成可视化的监控页面。

**核心目的：** 帮助投资者监控 LOF 基金的溢价/折价机会，识别套利窗口。

## 🌊 核心数据流：动静结合的 "魔法"

本系统巧妙利用 SQLite 数据库的 `ON CONFLICT DO UPDATE` 和 `COALESCE` 特性，实现了**实时流（Stream）与历史批处理（Batch）的大一统**，将“盘中实时变化”与“盘后历史归档”完美融合在同一张 `fund_history` 表中，且**永远不会撑爆数据库**。

完整的时间线与数据流转过程如下：

### 1. 盘中时段（09:30 - 15:00）：实时轮询与覆写
* `01_rt_lof` 和 `02_rt_index` 就像不知疲倦的粉刷匠。每隔 5 秒钟，它们就会将获取到的最新现价（Price）和成交量写入数据库。
* **不撑爆数据库的秘密**：写入时使用了 `ON CONFLICT(date, fund_code) DO UPDATE`。这意味着每一只基金每天永远只有 1 行数据，每 5 秒的高频轮询实际上只是在不断**覆写**今天这一行。
* Web 展示端 (`00_web_server`) 读取这一行时，拿到的就是实时的盘中价，并据此计算出实时折溢价。

### 2. 收盘时刻（15:00 定格）：蜕变为收盘价
* 下午 3 点停盘后，接口数据不再跳动。`01_rt` 和 `02_rt` 最后一次刷写进数据库的数据就永远定格在了那里。
* **此时，高频跳动的“盘中价”就自然而然地蜕变为了“今日最终收盘价”**。这最后一帧数据被完美留存，作为接下来计算静态估值的基础底座。

### 3. 盘后时段（晚上）：数据大一统与精修
* 运行 `03_eod_nav` 数据大一统更新器。它包含两个协同工作的爬虫，利用 SQL 的 `COALESCE(新值, 旧值)` 语法完美完成盘后精修：
    * **新浪历史爬虫**：获取当天的官方日线收盘价。它会用官方收盘价安全覆盖掉白天定格的盘中价（消除盘中最后一秒可能存在的微小毛刺，确保历史报表 100% 精准）。
    * **东财净值爬虫**：获取基金公司刚公布的今日官方净值 (NAV)。由于它在更新 SQL 时对 price 字段传入了 `NULL`，`COALESCE(NULL, fund_history.price)` 机制保证了**它会完美保留刚才的收盘价，绝不会将其抹除，仅仅只是把为空的 `nav` 字段填补上**。
* 运行 `05_eod_valuation`，根据当日指数的涨跌幅和昨日净值，精准计算出今日的**静态估值**存入数据库。

**总结：**
白天 `01/02` 占坑画草图 ➔ 下午定格收盘价 ➔ 晚上 `03` 填入官方净值 ➔ `05` 算出最终估值。整个生命周期天衣无缝，前端 `00` 只需极简读取！

## 🗄️ 数据库设计哲学：“基金大一统，指数快慢分离”

在底层存储设计上，本系统针对“LOF基金”和“指数”的不同业务属性，采取了两种截然不同的建表策略，这是非常经典的架构教学案例：

### 1. 基金大一统：`fund_history`
* **业务痛点**：基金同时拥有“盘中被基民炒作的交易现价 (Price)”和“盘后基金公司公布的真实净值 (NAV)”。
* **解决之道**：将实时流与历史归档合并为一张大表（以 `date` 和 `fund_code` 为联合主键）。白天，程序将它当作实时表，高频覆写当天的现价；晚上，程序将它当作历史表，利用 SQL 填补官方净值。一行数据完美融合了“今日最终收盘价”与“今日官方净值”，方便前端一键 `SELECT` 极速渲染。

### 2. 指数快慢分离：`index_realtime_quotes` & `index_history`
* **业务痛点**：指数没有“净值”概念，但前端需要“极速读取当前实时涨跌幅”，而后端在算估值时需要“大批量查询过去100天的历史K线”来寻找昨日基准点。如果把高频刷新的实时快照硬塞进日线历史表中，会导致前端查询变得极为臃肿。
* **解决之道**：为了极致的查询速度，将指数数据拆分为快、慢两张表！
  * **快表 (`index_realtime_quotes`)**：本质上是一个类似于 Redis 的超高速 Key-Value 缓存表。永远只保留全市场指数的**最新一行快照**。专供 Web 看板 (`009`) 闪电般读取，是页面实现“毫秒级”秒开的秘密武器。
  * **慢表 (`index_history`)**：历史日线 K 线表。专供静态估值引擎 (`05_eod_valuation`) 在盘后执行复杂的历史回溯与基准点对齐运算。
  
快慢分离、动静结合，互不干扰，将 SQLite 的读写性能压榨到了极致！

## 💡 核心业务逻辑揭秘：跨市场节假日错位的“双标”处理

在跨境 ETF/LOF 套利中，经常会遇到 A 股放假但海外市场（如港股/美股）开盘的情况（如五一长假、春节）。本系统针对这种情况实现了极其硬核的**“双标”处理逻辑**：

### 1. 后台算账：必须使用“累计涨跌幅” (例如 3.46%)
* **场景**：假设 A 股 4月30日 收盘后放假，而港股在 5月1日-5日 期间累计上涨了 3.46%。
* **逻辑**：此时 LOF 基金公布的最新官方净值（NAV）永远停留在 4月30日。后端 `05_eod_valuation` 静态估值引擎在回溯时，会精准提取 4月30日 的指数点位与今日点位做除法，得出 3.46% 的累计涨幅来乘以老净值。**唯有如此跨越假期缝隙，算出的今日估值才是 100% 准确的！**

### 2. 前台看盘：必须顺应直觉展示“单日涨跌幅” (例如 1.11%)
* **场景**：用户打开看盘软件，恒生指数今天的官方标定涨幅就是 1.11%。如果看板显示 3.46%，会造成极大的视觉误导。
* **逻辑**：前端轮询引擎 `02_rt_index` 在处理港股和跨市场指数时，强制交由东财 API 获取官方标准的单日涨幅（`f170` 字段）并存入快照表供前端直接渲染。

**总结：后台算账用累计（追求绝对精准），前台看盘用单日（顺应人类直觉）。底层数据架构通过历史 K 线表与实时快照表的物理隔离，完美支撑了这一复杂的业务场景！**

## � 快速开始

### 1. 环境准备
```bash
pip install pandas requests flask pytdx cachetools
```

### 2. 启动实时系统 (盘中必须开启的三个进程)
建议打开三个独立的终端窗口，分别运行：

```bash
# 窗口 1：启动纯净 Web 看板服务 (访问 http://127.0.0.1:5003)
python 00_web_server.py

# 窗口 2：启动 LOF 盘中实时获取引擎 (TDX+新浪)
python 01_rt_lof.py

# 窗口 3：启动指数盘中实时获取引擎 (TDX+东财)
python 02_rt_index.py
```

### 3. 盘后批处理维护 (每天 15:00 之后运行)
```bash
# 更新官方净值与配置 (推荐在基金公司公布净值后的晚间运行)
python jsl001_data_updater.py

# 计算当日最终的静态估值与误差 (可在 15:05 运行一次锁定)
python jsl005_update_static_valuation.py
```

## 📝 配置说明

`fund_list.csv` 配置字段：
- `分类`: 基金类别（指数LOF、T+0 QDII等）
- `代码`: 基金代码
- `名称`: 基金名称
- `仓位`: 指数仓位比例（如95%、90%）
- `相关指数`: 关联的指数名称
- `指数代码`: 东财格式的指数代码
- `申购费/赎回费`: 申赎费率
- `申购状态/赎回状态`: 开放/暂停/限额

## 🌟 架构演进总结 (v2.0)

本项目经历了从单体架构到 **MVC 微服务架构** 的重大重构：
- **完全前后端分离**：将 Web 看板 (`00_web`) 彻底剥离出网络请求逻辑，只负责极速读取本地数据库并展示，杜绝假死卡顿。
- **独立后台轮询矩阵**：新建 `01_rt` (LOF 轮询) 和 `02_rt` (指数轮询) 守护进程，专门负责盘中向数据库每 5 秒高频注入最新价格。
- **攻克反爬死锁**：利用东财 `Connection: close` + 随机有效边缘节点，彻底解决频繁请求带来的 `RemoteDisconnected` 死锁问题。
- **融合通达信引擎**：全面引入 `tqcenter` 官方接口库！历史 A 股 K 线、盘中实时 A 股指数和 LOF 盘口数据，现已优先利用本地通达信实现**极速、永不封禁**的数据获取，并将新浪与东财接口作为高可用降级备用方案。
- **打通 SQLite 底层闭环**：通过 `ON CONFLICT DO UPDATE` 和 `COALESCE` 技术，将高速瞬息万变的流式行情与盘后批量的官方归档，完美地熔入了一张 `fund_history` 表中，不冗余、不覆盖。
- **文件重命名规范化**：摒弃杂乱无章的旧前缀，采用 `rt` (实时) 和 `eod` (盘后) 加上直观的 `00~05` 数字序号，操作顺序一目了然，极大地降低了新手的学习与维护门槛。
